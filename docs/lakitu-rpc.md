# lakitu JSON-RPC reference

This file is generated by `.claude/skills/drift-docs/scripts/dump_rpc.sh`. Do not edit by hand — run the drift-docs skill to regenerate.

lakitu is the server half of drift. drift speaks to it over plain SSH using a one-shot JSON-RPC 2.0 protocol: one request in on stdin, one response out on stdout, SSH exits. State lives on the circuit under `~/.drift/garage/`.

---

## Transport

- **Framing**: one JSON object per invocation, newline-terminated. No batches, no notifications (every request must carry an `id`).
- **Params**: named params only — positional arrays are rejected at decode time.
- **Version**: `jsonrpc: "2.0"` required on both request and response.
- **Trigger**: `ssh <circuit> lakitu rpc < request.json` (the `rpc` subcommand reads stdin, dispatches, writes one response, exits).

### Request

```json
{
  "jsonrpc": "2.0",
  "method": "kart.list",
  "params": {},
  "id": 1
}
```

### Response (success)

```json
{
  "jsonrpc": "2.0",
  "result": { "...": "method-specific" },
  "id": 1
}
```

### Response (error)

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": 3,
    "message": "kart \"foo\" not found",
    "data": { "type": "kart_not_found", "kart": "foo" }
  },
  "id": 1
}
```

See `internal/wire/wire.go` for the exact Request/Response Go types and the decoder's validation rules.

## Version compatibility

Every drift invocation probes `server.version` first. The response carries a semver `version` string and an integer `api` field that bumps on breaking wire changes. drift's policy:

| mismatch          | behavior       |
|-------------------|----------------|
| major version     | error, abort   |
| minor version     | warn to stderr |
| patch version     | silent         |
| `api` mismatch    | error, abort   |

Bypass during upgrades with `drift --skip-version-check …`.

`server.info` is the richer setup-time variant (same version fields plus circuit identity and default character). Reserved for `drift init` and `drift circuit add`; it is not on the hot path.

## Errors

Errors serialize as the standard JSON-RPC error object with `code`, `message`, and an optional `data` payload. Exit codes on the human CLI path mirror `code`.

```text
  0 success · 2 user error · 3 not-found · 4 conflict
```

Clients should branch on `data.type` rather than `code` — the type is a stable snake_case identifier, whereas `code` is coarse. Current types:

```text
  internal_error               TypeInternalError
  invalid_name                 TypeInvalidName
  invalid_flag                 TypeInvalidFlag
  mutually_exclusive_flags     TypeMutuallyExclusive
  kart_not_found               TypeKartNotFound
  circuit_not_found            TypeCircuitNotFound
  character_not_found          TypeCharacterNotFound
  chest_entry_not_found        TypeChestEntryNotFound
  name_collision               TypeNameCollision
  stale_kart                   TypeStaleKart
  already_enabled              TypeAlreadyEnabled
  devpod_up_failed             TypeDevpodUpFailed
  devpod_ssh_failed            TypeDevpodSSHFailed
  devpod_unreachable           TypeDevpodUnreachable
  chest_backend_denied         TypeChestBackendDenied
  garage_write_denied          TypeGarageWriteDenied
  systemd_denied               TypeSystemdDenied
```

Some handlers attach extra context under well-known `data` keys (see `internal/rpcerr/rpcerr.go`):

- `devpod_stderr` — captured devpod stderr tail on `devpod_*` errors
- `devpod_stdout` — captured devpod stdout (devpod relays most real failure messages here, not stderr)

Additional arbitrary keys may appear per error type (e.g. `kart_not_found` attaches `kart` and `circuit`). Clients should tolerate unknown keys.
